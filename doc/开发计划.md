# WinAudioBridge 开发计划

## 1. 开发目标

当前阶段先完成 Windows 端 MVP 基础框架，重点落在以下能力：

- 应用启动后驻留系统托盘
- 托盘菜单支持“设置”“退出”
- 设置页支持配置编码、采样率、声道数、Buffer 大小
- 配置能够本地持久化
- 为后续音频采集、ADB 通信、推流控制预留结构

在音频链路稳定后，新增下一阶段增强目标：

- Android 端远程查看和控制 Windows 总音量
- Android 端远程查看和控制 Windows 各应用音量
- Android 端展示 Windows 应用名称与图标

## 2. 迭代原则

- 先搭框架，再接核心音频链路
- 每一轮开发结束必须更新开发状态文档
- 每次修改后进行构建验证
- 优先保证结构清晰、可持续扩展

## 3. 迭代拆分

### 第 1 轮：桌面端基础框架

目标：完成托盘、设置窗口、配置持久化。

交付内容：

- `App` 启动改为托盘常驻
- 托盘菜单：设置、退出
- 设置窗口
- 设置模型与本地 JSON 持久化
- 主窗口调整为状态页或占位页

验收标准：

- 启动后可以在系统托盘看到应用图标
- 点击“设置”可打开设置窗口
- 修改设置并保存后可持久化
- 点击“退出”可正常关闭进程

### 第 2 轮：Windows 端推流骨架

目标：完成 Windows 端服务层骨架。

交付内容：

- 基于 `AdvancedSharpAdbClient` 的 `AdbService`
- 设置页展示已连接 Android 设备列表
- 设置页展示目标 Android 音频应用运行状态
- `AudioCaptureService` 占位
- `AudioTransportService` 占位
- `StreamingCoordinator`
- 主界面展示配置与当前状态

验收标准：

- 应用内可显示当前配置
- 设置页可刷新当前连接 Android 设备
- 可识别目标 Android 音频应用是否在设备上运行
- 推流服务对象可被初始化和调度
- 关键服务接口稳定

### 第 3 轮：ADB 通信链路接入

目标：打通 ADB 转发和 TCP 连接。

交付内容：

- 执行 `adb devices`
- 执行 `adb forward`
- 建立 TCP Client
- `SessionInit` 协议发送

验收标准：

- 能检测设备
- 能建立本地 TCP 连接
- 能按协议发送初始化消息

### 第 4 轮：音频采集与 PCM 推流

目标：实现 Windows 系统音频采集并发送。

交付内容：

- 集成 `NAudio`
- `WasapiLoopbackCapture`
- 音频切帧
- `AudioFrame` 发送

验收标准：

- 可采集系统音频
- 可按设置项发送 PCM 帧

### 第 5 轮：稳定性与体验增强

目标：优化异常处理、日志和配置联动。

交付内容：

- 自动重连
- 日志系统
- 参数变更触发链路重建
- 更多托盘快捷操作

### 第 6 轮：Windows 远程音量控制

目标：实现 Android 对 Windows 总音量和应用音量的远程查看与控制。

交付内容：

- Windows 主音量读取、设置与静音控制
- Windows 应用音频会话枚举
- 单应用音量读取、设置与静音控制
- 应用名称生成与图标提取
- 双向控制协议扩展
- Android 音量控制页面与图标展示

验收标准：

- Android 端可显示 Windows 当前总音量
- Android 端可显示可控应用名称、图标和音量
- Android 端调整总音量后 Windows 立即生效
- Android 端调整单应用音量后目标应用立即生效
- Windows 本地改动音量后 Android 可同步刷新

## 4. 当前执行顺序

当前先执行：**第 1 轮：桌面端基础框架**。

- 当前完成内容
- 当前阻塞项
- 下一轮计划
- 最近一次验证结果
