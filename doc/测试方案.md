# WinAudioBridge 测试方案

## 1. 文档目标

本文档用于明确 WinAudioBridge 当前阶段的测试策略，避免 Windows 端与 Android 端重复构造对端 Mock，降低测试维护成本，并保证协议与关键业务逻辑具备可持续回归能力。

## 2. 总体原则

本项目测试采用“**契约测试 + 本地纯逻辑单元测试 + 少量跨端集成测试**”的分层策略。

核心原则如下：

- 不对 Windows 端和 Android 端重复编写“互相 Mock 对端”的测试
- 协议样本由两端共享，保证协议语义一致
- 单元测试优先覆盖纯逻辑、纯数据转换、选择策略与配置归一化逻辑
- 真正依赖设备、ADB、Socket、音量会话、`AudioTrack` 的链路，仅保留少量集成/联调用例

## 3. 分层测试策略

### 3.1 协议契约测试

目标：验证协议结构和字段语义一致。

建议覆盖：

- `SessionInit`
- `AudioFrame`
- `VolumeCatalogSnapshot`
- `CommandAck`
- 非法协议头
- 非法 `PayloadLength`
- 未知消息类型

建议方式：

- 建立共享协议样本目录
- Windows 与 Android 分别对同一批样本进行解析和断言

### 3.2 Windows 端单元测试

目标：只验证 Windows 本地纯逻辑，不依赖 Android 端存在。

首批建议覆盖：

- `SettingsService` 配置归一化逻辑
- `StreamingCoordinator` 目标设备选择逻辑
- `VolumeIconService` 图标键/摘要逻辑

不建议在单元测试中直接覆盖：

- ADB 真连接
- WASAPI 真采集
- Core Audio 真音量控制
- 真实 TCP 连接

### 3.3 Android 端单元测试

目标：只验证 Android 本地解析、状态合并与 ViewModel 行为。

建议覆盖：

- 协议 JSON/二进制解析
- `PlaybackStateRepository` 对目录、回执、增量消息的合并逻辑
- icon 保留与覆盖逻辑

当前已落地：

- `WindowsVolumeJsonCodec` 请求构造与 JSON 解析测试
- `ProtocolReader` 二进制协议包解析测试
- 非法协议头校验测试

### 3.4 少量跨端集成测试

目标：验证关键真实链路。

建议仅保留：

- 建链成功
- `SessionInit` 成功
- 音量目录拉取成功
- 主音量调整成功
- 应用音量调整成功
- 断线自动重连成功

## 4. 当前实施顺序

当前优先级如下：

1. 先补 Windows 端纯逻辑单元测试
2. 再补 Android 端协议/状态单元测试
3. 最后补少量真机联调清单

当前执行结果：

- Windows 首批纯逻辑单元测试已完成并通过
- Android 首批协议解析单元测试已完成并通过

## 5. 当前已落地的首批 Windows 测试范围

本轮测试先覆盖以下内容：

- `SettingsService.Normalize()`
- `StreamingCoordinator.SelectTargetDevice()`
- `VolumeIconService.CreateIconKey()` / `GetIconHash()`

这些测试具有以下特点：

- 无需 Android 端 Mock 服务
- 无需真实 ADB
- 无需真实 Socket
- 执行快、稳定、适合持续集成

## 6. 后续扩展建议

后续可继续增加：

- 共享协议样本目录
- `AudioTransportService` 协议包编解码测试
- `WindowsVolumeService` DTO 映射测试（通过抽象会话快照输入）
- Android 端 `PlaybackStateRepository` 状态合并测试
- Android 端 ViewModel 行为测试

## 7. 结论

本项目不建议让 Windows 与 Android 两端都重复构造对端 Mock。

推荐做法是：

- 协议一致性测一次
- 本地纯逻辑分别测一次
- 真链路少量测一次

该策略能在保证质量的同时，显著降低测试冗余与维护成本。
