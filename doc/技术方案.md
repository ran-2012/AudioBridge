# WinAudioBridge 技术方案

## 1. 项目目标

WinAudioBridge 旨在实现“将 Windows 电脑的系统音频通过 ADB 转发到 Android 设备播放”的能力，整体体验类似 AudioRelay，但当前阶段聚焦于 **USB/ADB 有线传输** 场景。

项目目标如下：

- 采集 Windows 当前系统播放音频。
- 通过 ADB 端口转发建立 Windows 与 Android 之间的稳定数据通道。
- 在 Android 端实时播放来自 Windows 的音频流。
- 在 Android 端远程查看并控制 Windows 总音量与各应用音量。
- 在 Android 端展示 Windows 音频会话对应的应用名称与图标。
- Windows 客户端启动后常驻系统托盘，减少主窗口占用。
- 通过设置界面配置编码、采样率、声道数、Buffer 大小等关键音频参数。
- 通过设置界面查看当前已连接的 Android 设备。
- 在设置界面显示哪些已连接设备上已运行 Android 音频接收应用。
- 优先完成可运行的 MVP，再逐步优化延迟、稳定性与静音体验。

## 2. 关键技术约束

根据前期调研，项目存在以下关键限制：

1. **无法使用纯 C# 编写 Windows 虚拟音频驱动。**
	 Windows 虚拟声卡/虚拟输出设备本质上属于驱动开发范畴，需要基于 C/C++ 和 WDK 实现，复杂度高，不适合作为当前项目首阶段目标。

2. **Windows 音频采集推荐采用 WASAPI Loopback。**
	 在不开发驱动的前提下，最可行的方案是捕获默认输出设备正在播放的系统音频。

3. **ADB 可替代底层 USB 通讯开发。**
	 借助 `adb forward tcp:5000 tcp:5000`，可以直接使用标准 TCP Socket 在 PC 与 Android 间传输数据，无需自行实现 USB 通讯协议。

4. **首版优先使用 PCM 直传。**
	 先确保端到端链路打通，再考虑引入 Opus 压缩降低带宽与抖动风险。

## 3. 技术路线选型

### 3.1 Windows 音频采集方案对比

#### 方案 A：WASAPI Loopback Capture（推荐）

通过 NAudio 封装的 `WasapiLoopbackCapture` 捕获默认播放设备输出的系统音频。

**优点：**

- 完全基于 C# 实现，开发门槛低。
- 无需编写驱动，适合快速落地。
- 生态成熟，便于调试与扩展。

**缺点：**

- 采集结果依赖默认输出设备。
- 系统静音、音量变化可能影响采集效果。
- 无法天然实现“PC 本地完全无声但手机有声”的最佳体验。

#### 方案 B：第三方虚拟音频线（备选增强）

要求用户安装 VB-Audio Cable 等第三方虚拟声卡，将其设置为默认输出设备，再由程序进行采集。

**优点：**

- 可实现 PC 侧静音、Android 侧播放的更好体验。
- 无需自研驱动。

**缺点：**

- 增加用户安装和配置成本。
- 对产品易用性有一定影响。

#### 方案 C：自研 Windows 虚拟音频驱动（不建议现阶段采用）

基于 WDK 与 Microsoft `sysvad` 示例开发自有虚拟播放设备。

**优点：**

- 可实现完整、专业的产品形态。
- 可构建独立品牌化的虚拟输出设备。

**缺点：**

- 驱动开发成本极高。
- 需掌握 C/C++、WDK、驱动签名、安装分发等复杂链路。
- 不适合作为 MVP 阶段技术路线。

### 3.2 最终选型

本项目推荐采用：

- **首期 MVP：方案 A（WASAPI Loopback）**
- **中期增强：支持方案 B（VB-Cable 等第三方虚拟音频线）**
- **长期规划：评估自研驱动方案，但不纳入当前开发范围**

### 3.3 ADB 开发库选型

Windows 端 ADB 能力建议优先采用 **AdvancedSharpAdbClient**。

采用原因：

- 可直接在 .NET 中获取设备列表。
- 可执行设备 Shell 命令，用于检测 Android 应用是否正在运行。
- 可用于后续端口转发、应用启动、安装与状态查询。
- 比纯命令行输出解析更便于维护和测试。

## 4. 总体架构设计

系统整体分为三部分：

1. **Windows 桌面端（WPF + C#）**
2. **ADB 转发链路**
3. **Android 播放端**

### 4.1 架构流程

1. Windows 桌面端启动后检测 ADB 环境与设备连接状态。
2. 调用 ADB 建立端口转发：本机 `127.0.0.1:5000` -> Android `tcp:5000`。
3. Windows 端使用 NAudio 的 `WasapiLoopbackCapture` 采集系统音频 PCM 数据。
4. 采集到的数据通过 `TcpClient` 写入本地 `127.0.0.1:5000`。
5. ADB 将该数据流转发到 Android 设备。
6. Android 端 `ServerSocket` 接收数据流，并交由 `AudioTrack` 播放。
7. Android 端通过同一通信链路或独立控制链路请求 Windows 音量目录并下发音量控制命令。
8. Windows 端通过 Core Audio Session API 回传主音量、应用会话列表、名称与图标信息。

### 4.2 模块划分

#### Windows 端模块

- **UI 模块（WPF）**
	- 系统托盘图标与右键菜单
	- 设置窗口
	- 设备连接状态显示
	- 已连接 Android 设备列表展示
	- Android 音频应用运行状态展示
	- 启停控制
	- 日志输出
	- 音频格式/码率配置

- **系统托盘模块**
	- 应用启动后隐藏到托盘
	- 托盘菜单包含“设置”“退出”
	- 支持双击托盘图标打开设置或主状态窗口

- **配置管理模块**
	- 保存编码、采样率、声道数、Buffer 大小
	- 保存 Android 应用包名
	- 启动时加载本地配置
	- 配置变更后通知采集与传输模块重建链路

- **ADB 管理模块**
	- 检测 `adb` 是否可用
	- 检测设备是否在线
	- 枚举已连接 Android 设备
	- 检测目标 Android 应用是否正在运行
	- 执行端口转发
	- 异常恢复与重连

- **音频采集模块**
	- 初始化 `WasapiLoopbackCapture`
	- 获取采样率、声道数、位深/浮点格式
	- 数据回调处理

- **Windows 音量控制模块**
	- 读取和设置系统主音量
	- 枚举应用音频会话
	- 读取和设置单应用音量/静音
	- 监听音量变化并向 Android 回推最新状态
	- 提取应用名称与图标

- **音频发送模块**
	- 建立 TCP 连接
	- 将 PCM 数据写入网络流
	- 缓冲与异常处理

- **控制协议模块**
	- 处理 Android 发起的音量目录请求
	- 处理主音量与应用音量控制命令
	- 发送图标和音量状态快照/增量更新

- **编码扩展模块（后续）**
	- PCM -> Opus 编码
	- 包头封装
	- 码率与帧长控制

#### Android 端模块

- **连接监听模块**
	- 启动 `ServerSocket`
	- 接收来自 ADB 转发的 Socket 连接

- **远程控制界面模块**
	- 展示 Windows 总音量
	- 展示应用音量列表、应用名称与图标
	- 发送音量调整和静音命令

- **音频解码/格式适配模块**
	- MVP 阶段直接处理 PCM
	- 后续支持 Opus 解码

- **播放模块**
	- 使用 `AudioTrack` 初始化输出
	- 写入音频帧并保持稳定播放

- **缓冲控制模块**
	- 防止 underrun
	- 统计播放延迟与丢包情况

- **音量状态仓库模块**
	- 保存 Windows 音量目录快照
	- 管理图标缓存
	- 处理增量更新与手动刷新

### 4.3 桌面端交互设计

Windows 客户端推荐采用“**托盘常驻 + 设置窗口**”模式，而非传统主窗口常驻模式。

#### 启动行为

- 应用启动后初始化核心服务。
- 默认不强依赖主窗口常驻显示，可直接驻留系统托盘。
- 首次启动可弹出一次引导或状态窗口，后续最小化到托盘。

#### 托盘菜单

托盘右键菜单至少包含：

- **设置**：打开参数配置窗口。
- **退出**：停止音频采集、断开 Socket、清理 ADB 转发后退出程序。

后续可扩展：

- 开始推流
- 停止推流
- 查看状态
- 重新连接设备

## 5. 数据链路设计

### 5.1 MVP 阶段数据格式

首期建议使用 **PCM 原始数据流**，减少编码复杂度。

建议统一格式：

- 采样率：`48000 Hz`
- 声道：`2`（立体声）
- 编码：优先兼容 NAudio 实际输出格式
	- 常见为 `IEEE Float 32-bit`
	- 如 Android 端兼容性不足，可在 Windows 端转换为 `PCM 16-bit`

### 5.2 可配置参数设计

根据新增需求，Windows 客户端设置页需要支持以下关键参数：

- **编码格式**
	- `PCM16`
	- `Float32`
	- `Opus`（后续版本）

- **采样率**
	- `44100 Hz`
	- `48000 Hz`（推荐默认）
	- 其他扩展值可按设备能力开放

- **声道数**
	- `1`（单声道）
	- `2`（立体声，推荐默认）

- **Buffer 大小**
	- 建议以毫秒为配置单位，例如 `20ms`、`40ms`、`60ms`、`100ms`
	- 内部换算为每帧样本数和字节数

- **Android 应用包名**
	- 用于识别哪些设备已经安装并运行接收端应用
	- 用于后续启动应用、拉起服务和状态检测

配置项设计建议：

- 首版采用“**预设选项 + 下拉框**”而不是自由输入，降低非法参数风险。
- 配置变更后要求重新初始化采集链路与播放链路。
- 设置页提供“恢复默认值”能力。
- 设置页提供“刷新设备”能力，实时查看连接设备及应用运行状态。

### 5.3 建议的传输协议

虽然最简实现可以直接裸传 PCM，但为了便于扩展，建议设计轻量级包头：

#### 会话头

在正式发送音频前，Windows 端先发送一次音频格式信息：

- 协议版本
- 编码类型（PCM16 / Float32 / Opus）
- 采样率
- 声道数
- 每帧样本位数

#### 音频帧

每个数据包包含：

- 帧长度
- 时间戳（可选）
- 音频负载数据

这样做的好处：

- 后续切换编码格式时无需重写整体协议。
- Android 端可根据包头动态初始化 `AudioTrack`。
- 更容易定位网络与播放问题。

## 6. Windows 端实现方案

### 6.1 技术栈

- .NET 9 WPF
- NAudio
- AdvancedSharpAdbClient
- `System.Net.Sockets`
- `System.Diagnostics.Process`

### 6.2 关键流程

#### 1）ADB 初始化

应用启动或点击“开始推流”时：

- 通过 `AdvancedSharpAdbClient` 启动或连接 ADB Server
- 获取已连接 Android 设备列表
- 检测目标 Android 应用是否正在运行
- 执行 `adb forward tcp:5000 tcp:5000` 或等价 Forward 能力
- 若失败，在 UI 中提示错误原因

#### 2）建立 Socket 连接

- Windows 端连接 `127.0.0.1:5000`
- 获取 `NetworkStream`

#### 3）启动系统音频采集

- 创建 `WasapiLoopbackCapture`
- 订阅 `DataAvailable`
- 在回调中获取 PCM 数据

#### 4）数据发送

- 将采集数据按帧写入 `NetworkStream`
- 避免在 UI 线程中执行发送逻辑
- 增加发送状态监控与自动停止机制

#### 5）托盘初始化

- 应用启动后创建 `NotifyIcon`
- 注册托盘菜单：`设置`、`退出`
- 点击“设置”打开配置窗口
- 点击“退出”执行优雅关闭流程

#### 6）配置加载与热更新

- 启动时读取本地配置文件
- 若配置发生变更，则按需重建编码器、采集器和发送链路
- 对于不能热切换的参数，提示“应用后重启推流”

### 6.3 WPF 客户端建议界面

建议 Windows 客户端采用“托盘为主、设置窗口为辅”的结构。

#### 托盘菜单

- 设置
- 退出

#### 设置窗口包含：

- ADB 状态
- Android 设备状态
- 已连接设备列表
- Android 音频应用运行状态
- 当前 Windows 主音量与远程控制链路状态
- 当前采样率/声道/编码格式
- 编码格式选择
- 采样率选择
- 声道数选择
- Buffer 大小选择
- Android 应用包名配置
- 开始/停止按钮
- 日志面板
- “使用虚拟音频线”选项说明

后续可扩展展示：

- 当前同步到 Android 的应用音量会话数量
- 最近一次音量目录同步时间
- 图标缓存命中情况

设置窗口交互建议：

- “保存并应用”按钮：保存配置并尝试重建当前音频链路。
- “取消”按钮：关闭窗口，不修改当前配置。
- “恢复默认值”按钮：回退推荐参数。
- 参数发生变化但当前正在推流时，提示用户是否立即重启推流。

### 6.4 MVP 推荐类设计

建议初步拆分为以下类：

- `AdbService`
	- 负责连接 ADB Server、检测设备、识别应用运行状态、建立转发

- `AudioCaptureService`
	- 负责初始化 NAudio 并输出音频数据事件

- `AudioTransportService`
	- 负责建立 TCP 连接、发送数据、断线处理

- `WindowsVolumeService`
	- 负责 Windows 主音量和应用音量会话的读取、控制与监听

- `VolumeIconService`
	- 负责提取应用图标并转换为协议可传输格式

- `StreamingCoordinator`
	- 负责串联 ADB、采集、发送三者的启动顺序

- `MainViewModel`
	- 负责界面状态与用户交互

- `TrayService`
	- 负责托盘图标、菜单注册、菜单事件分发

- `SettingsService`
	- 负责配置读取、保存、默认值管理

- `SettingsViewModel`
	- 负责设置界面参数绑定与校验

- `AndroidDeviceInfo`
	- 负责承载设备型号、序列号、连接状态、应用运行状态等信息

### 6.5 配置持久化方案

建议将用户设置持久化到本地配置文件，例如：

- `settings.json`

建议保存字段：

- `Encoding`
- `SampleRate`
- `Channels`
- `BufferMilliseconds`
- `AndroidAppPackageName`
- `AutoStartStreaming`（可选）
- `StartMinimizedToTray`（可选，建议默认开启）

配置原则：

- 启动时自动加载。
- 配置文件缺失时自动生成默认配置。
- 对异常值进行校验和回退。
- 保存配置后立即刷新 UI 显示。

## 7. Android 端实现方案

### 7.1 技术栈建议

- Kotlin
- `ServerSocket`
- `AudioTrack`
- 后续可引入 `MediaCodec` 或 `libopus`

### 7.2 核心流程

1. 启动后台服务或前台服务。
2. 监听 `5000` 端口。
3. 接收 Windows 端传来的连接。
4. 读取协议头，初始化播放格式。
5. 持续读取音频帧并写入 `AudioTrack`。
6. 连接断开后进入等待重连状态。

### 7.3 播放参数要求

Android 端 `AudioTrack` 的参数必须与 Windows 端输出一致：

- 采样率一致
- 声道一致
- PCM 编码格式一致

如果 Windows 端输出 Float32，而 Android 端设备兼容性不佳，则建议在 Windows 端统一转换成 16-bit PCM，以提升通用性。

## 8. 压缩与性能优化方案

### 8.1 MVP 阶段

直接发送 PCM，优先保证：

- 正确性
- 稳定性
- 端到端低延迟

### 8.2 后续优化

引入 Opus 编码：

- Windows 端：使用 `Concentus` 将 PCM 编码为 Opus
- Android 端：使用原生解码能力或集成 `libopus`

优化收益：

- 显著降低传输带宽
- 减少高负载下卡顿概率
- 为后续无线传输预留空间

### 8.3 缓冲控制建议

- Windows 端按固定时长切帧，例如 10ms / 20ms
- Android 端维护小型环形缓冲区
- 增加播放统计：平均延迟、缓冲深度、断流次数

## 9. 异常处理与稳定性设计

需要重点处理以下异常：

### 9.1 ADB 异常

- 未安装 ADB
- 未开启 USB 调试
- 设备未授权
- 设备掉线
- 端口转发失败

### 9.2 音频异常

- 默认输出设备切换
- 采集设备不可用
- 音频格式变化
- 回调线程阻塞

### 9.3 网络流异常

- Socket 连接中断
- 写入失败
- Android 端服务未启动

### 9.4 恢复策略

- 自动停止采集并释放资源
- 提示用户重新连接设备
- 支持“一键重试”
- 记录日志便于问题定位

## 10. 安全性与权限说明

### Windows 端

- 需要本地执行 `adb.exe`
- 需要访问音频设备

### Android 端

- 需要前台服务权限（若后台持续播放）
- 需要网络 Socket 能力
- 若涉及开机自启、常驻服务，需额外评估系统兼容性

## 11. 开发阶段规划

### 第一阶段：MVP 打通链路

目标：实现“Windows 系统声音 -> Android 播放”最小闭环。

范围：

- WPF 界面基础搭建
- ADB 检测与端口转发
- NAudio Loopback 捕获
- TCP PCM 推流
- Android `AudioTrack` 播放

验收标准：

- Android 能稳定播放 Windows 系统音频
- 延迟可接受
- 连续播放 30 分钟无明显异常

### 第二阶段：增强稳定性

范围：

- 自动重连
- 日志系统
- 格式协商
- PCM16 兼容模式
- 缓冲优化
- 配置持久化
- 托盘交互完善

### 第三阶段：体验优化

范围：

- 支持 VB-Cable 等虚拟音频线
- 支持设备选择
- 支持采样率/编码配置
- 优化 UI 与状态提示
- 支持更多托盘快捷操作

### 第四阶段：高级能力

范围：

- Opus 编码压缩
- 无线传输预研
- 多设备支持
- 长期评估自研虚拟音频驱动可行性

## 12. 风险评估

### 高风险项

1. **WASAPI Loopback 受系统输出状态影响**
2. **不同 Android 设备对 Float PCM 支持不一致**
3. **长时间播放时缓冲控制不当可能出现爆音或卡顿**
4. **ADB 连接稳定性受 USB 线材、驱动和设备状态影响**

### 风险缓解措施

- MVP 阶段优先使用更通用的 PCM16 输出
- 引入协议头，降低格式不一致风险
- 增加日志与状态监控
- 中期引入虚拟音频线支持

## 13. 结论

基于当前技术约束，WinAudioBridge 最合理的实现路径是：

- **Windows 端使用 NAudio + WASAPI Loopback 捕获系统音频**
- **使用 ADB 端口转发承载音频数据传输**
- **Android 端使用 ServerSocket + AudioTrack 播放音频流**

该路线具备以下特点：

- 开发周期短
- 技术风险可控
- 完全符合当前 C# 技术栈能力边界
- 能快速完成 MVP 并为后续压缩、静音增强、无线化扩展打下基础

因此，建议项目按“**PCM 直传 MVP -> 稳定性增强 -> 压缩优化 -> 体验提升**”的路径推进实施。
